# 需求实施计划

## 重梳原则

- 本清单以 `requirements.md`、`design.md` 与当前仓库代码现状交叉核对后的结果为准。
- 当前仓库已经具备第一轮代码骨架和测试样例，距离“可联调、可上线、可运维”的目标还有明显差距。
- 只有已有代码与测试能直接证明的事项标记为 `[x]`；设计已明确但仍需落地的事项标记为 `[ ]`。

  - 已覆盖扩展层基础单元测试，对应当前仓库中的扩展层测试文件。

## B. Phase 1: 站内问答 MVP

### Phase 1 目标

- 形成“用户可在 BookStack 页面内提问并获得带引用回答”的最小可用闭环。
- 范围以页面正文为主，索引方式允许先采用手动全量触发。
- 当前策略采用“前端优先 + 低侵入挂载 + 暂缓真实模型”，先完成真实 BookStack 页面接入和闭环演示，再替换模型侧 stub。
- 该阶段完成后，允许继续保留真实增量索引、外部 API 和多租户治理为后续阶段任务。

### Phase 1 依赖

- 需要确认 BookStack 前端扩展挂载方式。
- 当前阶段允许继续使用 stub 推理，真实模型调用配置延后到 B2 前确认。
- 需要页面正文全量导入或全量重建索引入口。

- [ ] B1. 完成真实 BookStack 前端接入
  - 前置决策引用：`decision-h1-bookstack-extension.md`，按其中确认的挂载点、资源注入方式、样式隔离方式和状态持久化方式实施。
  - 将当前扩展层模块接入 BookStack 的实际扩展点或主题注入方案，落实设计中的页面侧边栏主入口与全局顶部次入口。
  - 完成侧边栏问答 UI、范围切换、引用跳转和失败态展示的真实页面集成。
  - 推荐实施顺序：先做主题模块骨架，再做顶部入口与页面挂载位，再做前端资源加载与状态恢复，最后接 AI Gateway 问答交互。
  - 子任务：
    - B1.1 创建 BookStack 主题模块骨架，包含 `bookstack-module.json`、`functions.php`、`views/`、`head/`、`public/bookstack-ai/`。
    - B1.2 在 `header-links-start` 首项接入全局顶部 AI 次入口，默认触发全局问答抽屉。
    - B1.3 使用 `ThemeEvents::THEME_REGISTER_VIEWS` 在 `pages.parts.show-sidebar-section-actions` 之后挂入页面级 AI 主入口。
    - B1.4 在 `base-body-end` 注入前端脚本，在模块 `head/` 注入样式与必要 head 元数据，确保脚本带 `nonce`。
    - B1.5 将当前仓库已有扩展层状态控制与查询客户端接入挂载根节点，打通抽屉开合、范围切换、提问提交、流式展示、引用跳转和失败态。
    - B1.6 验证 AI 服务异常时页面浏览、编辑、导航和原生侧栏保持可用，确保故障隔离满足 Requirement 1.4。
  - 需求映射：Requirement 1.1、1.3、2.1、2.2。
  - 验收口径：BookStack 页面内可见 AI 入口，可展开问答抽屉，可显示当前范围和引用链接。
  - 完成标准补充：顶部入口、页面主入口、脚本样式加载、状态恢复和 AI 不可用降级都具备可手工验证的结果。

- [ ] B2. 替换问答链路中的 stub 能力
  - 将 `Inference Adapter` 从当前 stub 输出替换为真实 LLM 调用。
  - 让同步回答与 `SSE` 流式回答共享同一套真实推理实现与错误处理。
  - 需求映射：Requirement 2.2、2.3、5.1、5.3、6.4。
  - 验收口径：回答内容不再包含 stub 文本，流式和非流式均可返回真实模型结果与结构化错误。

- [ ] B3. 打通最小可用索引链路
  - 增加手动全量索引入口，满足设计中 `Phase 1` 的“手动全量索引”目标。
  - 完成 BookStack 页面正文到标准化、切片、向量化、引用返回的闭环联调。
  - 在回答中稳定返回页面级或段落级引用。
  - 需求映射：Requirement 2.2、3.1、3.2、3.3。
  - 验收口径：选定一批测试页面执行全量索引后，问答可稳定命中这些页面并返回正确引用。

- [ ] B4. 补齐 Phase 1 验收测试
  - 前置决策引用：`decision-h1-bookstack-extension.md`，覆盖顶部入口、页面主入口、状态持久化与 AI 异常隔离验证。
  - 增加站内问答主路径端到端测试，覆盖提问、流式返回、引用展示和证据不足响应。
  - 验证 AI 服务异常时 BookStack 基础浏览与编辑能力保持可用，对应需求 1.4 和 Correctness Property 4。
  - 需求映射：Requirement 1.4、2.3、2.4、6.3。
  - 验收口径：具备可重复执行的 MVP 验收脚本或测试集。

- [ ] B5. Phase 1 退出检查点
  - 页面内主入口、真实回答、手动全量索引和引用展示已形成闭环。
  - BookStack Core 在 AI 不可用时仍保持基础浏览与编辑能力。
  - 进入下一阶段前，冻结 MVP 的交互和接口行为。

## C. Phase 2: 增量索引、权限过滤、审计与重试

### Phase 2 目标

- 把 MVP 提升为可持续运行的内部能力。
- 重点完成真实存储、真实队列、增量索引、权限约束和审计重试闭环。

### Phase 2 依赖

- Phase 1 已形成可演示的站内问答闭环。
- PostgreSQL、`pgvector`、Redis 和 PDF 解析服务具备可用运行环境。

- [ ] C1. 接入真实持久化与向量检索
  - 将内存版 repository 替换为 PostgreSQL 与 `pgvector` 实现。
  - 将当前检索逻辑接入真实 chunk 数据、权限范围、层级路径和语言过滤。
  - 子任务：先固化 repository 接口映射，再落库 schema 与 migration，再补查询性能基线。
  - 需求映射：Requirement 2.1、3.3、6.4、7.1、7.2。
  - 验收口径：重启服务后索引数据和审计数据仍可查询，检索命中来自真实数据库。

- [ ] C2. 接入真实索引任务队列
  - 将内存版 stream client 替换为 `Redis Streams consumer group` 实现。
  - 完成主流、重试流、死信流、幂等键和任务状态流转的真实消费链路。
  - 子任务：先实现生产者写入，再实现单消费者处理，再扩展到 consumer group 和重试死信。
  - 需求映射：Requirement 3.1、3.4、6.1、6.2。
  - 验收口径：重复事件不会重复索引，失败事件按预期进入重试或死信。

- [ ] C3. 完成增量索引与附件索引联调
  - 将页面创建、更新、移动、删除事件与索引任务真实打通。
  - 将 PDF 解析服务调用、页码定位、失败回写和重试能力接入真实运行链路。
  - 子任务：优先打通页面正文增量索引，其次补齐 PDF 附件链路。
  - 需求映射：Requirement 2.2、3.1、3.2、3.4、6.3。
  - 验收口径：页面修改后能触发索引更新，附件解析失败可见失败原因并可重试。

- [ ] C4. 完成权限与审计治理落地
  - 确保每次检索先做租户过滤，再做权限过滤，再做召回，对应 Correctness Property 1。
  - 落实问答审计、查询历史、失败原因、延迟与 token 用量查询接口。
  - 增加管理端任务重试入口。
  - 子任务：先固化过滤顺序，再接入审计落库，再补管理端查询与重试入口。
  - 需求映射：Requirement 2.1、2.4、5.4、6.1、6.2、7.1、7.2。
  - 验收口径：权限变更后历史 chunk 不会继续命中，审计记录可按租户查询。

- [ ] C5. 补齐 Phase 2 验收测试
  - 增加 BookStack 事件到索引入库的全链路集成测试。
  - 增加页面删除、权限变更、索引重建的回归测试。
  - 需求映射：Requirement 3.1、3.4、7.2、7.4。
  - 验收口径：Phase 2 主路径具备自动化回归测试。

- [ ] C6. Phase 2 退出检查点
  - 真实数据库、真实队列、增量索引、权限过滤、审计与重试能力已完成联调。
  - 内部问答链路具备持续运行条件，管理端具备基本排障抓手。
  - 进入下一阶段前，冻结内部接口与索引数据契约。

## D. Phase 3: 对外 RAG API、多租户隔离、限流与配额治理

### Phase 3 目标

- 在内部问答能力稳定后，对外开放受控的企业知识问答接口。
- 重点解决客户端身份、知识范围授权、租户隔离、限流配额和审计治理。

### Phase 3 依赖

- Phase 2 的真实检索、审计和权限能力已稳定。
- 已明确外部客户端接入方式、凭据生命周期和运维责任边界。

- [ ] D1. 完成外部 RAG API 产品化
  - 固化 `POST /v1/rag/query` 的同步与流式协议，并补齐 `retrieve` 与 `health-check` 能力。
  - 明确返回字段 `answer`、`citations`、`request_id`、`usage`、`latency_ms` 的兼容契约。
  - 子任务：先冻结请求响应 schema，再补文档与错误码，再补 SDK 或调用示例。
  - 需求映射：Requirement 4.1、4.2、4.3、6.3。
  - 验收口径：外部系统可基于稳定接口完成联调。

- [ ] D2. 完成 API Key 与授权范围治理
  - 将 `API Key`、租户隔离、书架或书籍白名单和客户端状态接入真实存储。
  - 落实外部客户端知识范围收缩与机器可读错误码。
  - 子任务：先建立客户端表与密钥存储规则，再实现作用域解析和禁用状态。
  - 需求映射：Requirement 4.1、4.4、5.3、7.1、7.2。
  - 验收口径：不同客户端只能访问被授权范围内的知识。

- [ ] D3. 完成限流、配额与外部审计
  - 实现客户端级限流、配额扣减和超限响应。
  - 将外部调用的审计记录纳入统一查询与保留期治理。
  - 子任务：先实现限流，再实现配额，再统一归档审计字段。
  - 需求映射：Requirement 4.4、5.4、6.1、6.2、7.3。
  - 验收口径：超限请求返回稳定错误码，外部调用可追踪到完整 request_id 链路。

- [ ] D4. 补齐 Phase 3 验收测试
  - 增加外部 API 的认证、白名单授权、限流和流式响应端到端测试。
  - 增加租户切换与跨租户不可见的隔离测试，对应需求 7.1、7.2。
  - 需求映射：Requirement 4.1、4.4、7.1、7.2。
  - 验收口径：外部能力具备上线前的自动化回归基础。

- [ ] D5. Phase 3 退出检查点
  - 外部客户端可通过受控凭据稳定调用问答接口。
  - 多租户隔离、范围授权、限流配额和外部审计具备上线约束能力。
  - 进入下一阶段前，冻结外部 API 契约和错误码集合。
